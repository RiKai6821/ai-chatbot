## 1. 一分钟项目概述

核心设计原则：**大脑在云端，设备只做瘦客户端**——设备不跑大模型，只负责"听/说/显示"，
通过 WiFi/HTTP 把音频交给服务器。好处：设备成本低（整机 <￥100）、可随时升级"大脑"。

```
设备端(ESP32-S3, C/ESP-IDF)                  云端(Python/FastAPI)
按键→I2S录音(DMA) ──WiFi/HTTP(WAV)──▶  /voice: STT→大模型(工具/RAG/记忆)→TTS
GC9A01圆屏表情 ◀─状态机─ I2S播放 ◀──(WAV)──
```

## 2. 我做了什么（两条线 + 亮点）

### 嵌入式线（`firmware-idf/`，ESP-IDF 纯 C）
- **BSP 分层 + 多应用**：把 WiFi / I2S 音频 / GC9A01 屏抽成可复用组件（`bsp_*`），
  上面 4 个应用（chat / audio / display / 整合语音助手）复用。
- **外设驱动 + DMA**：用 IDF v5 新版 `i2s_std` 驱动 INMP441 录音、MAX98357A 播放，
  双 I2S 端口、DMA 搬运；GC9A01 圆屏用 `esp_lcd`（SPI+DMA），**分带渲染省内存**。
- **RTOS 系统设计**：按键走 **GPIO 中断 → FreeRTOS 队列**，状态机任务跑"待机→听→想→说"，
  一轮对话 **HTTP 流式双向音频**（边录边传、边收边播），不缓存整段、不依赖 PSRAM。
- 工程化：CMake / Kconfig(menuconfig) / 组件管理器拉取依赖。

### Agent 线（根目录 Python）
- **接口**：`/chat`、`/chat/stream`(SSE 流式)、`/agent`(工具调用)、`/voice`(语音进出)；
  上下文压缩 + SQLite 会话持久化。
- **工具调用编排**：函数调用闭环（时间/天气/计算/知识检索），命令行与 HTTP 共用一套。
- **RAG 检索增强**：私有知识库向量化(text-embedding-v3) + 余弦检索，让助手据资料作答不瞎编。
- **评测框架**：用例集 + 工具选择/关键词/LLM 裁判三类判分，退出码接 CI。
- **可观测性**：每轮对话结构化 JSONL 日志（token、各工具耗时、延迟 p50/p95）。
- **高并发 + 容错**：全异步(AsyncOpenAI)；并发限流 + 指数退避重试 + 超时 + 边缘背压 429。
- **交付**：Dockerfile + compose 容器化；GitHub Actions CI（离线测试门禁 + 密钥门控真实评测）。

## 3. 难点与取舍（挑 2–3 个讲，STAR）

**① 设备内存有限，怎么播放/上传音频？**
不缓存整段。上行用 `esp_http_client_open` 声明总长后边录边 `write`；下行边 `read` 边推给
I2S 播放，并处理**跨数据块的奇数字节对齐**。屏幕用 240×40 **分带渲染**而非整屏帧缓冲。
→ 体现对受限资源的工程意识。

**② 异步下阻塞调用会拖垮事件循环。**
服务端改全异步后，STT/TTS（同步网络库）和 RAG 的 embedding 调用是阻塞的，直接 await 会
卡住整个 loop。用 `asyncio.to_thread` 把这些卸载到线程池。实测 5 并发墙钟 0.83s vs 串行
3.16s，**约 3.8x** 重叠。

**③ 上游模型会限流/超时，怎么保证服务不雪崩？**
四道防线：单次调用 `wait_for` 硬超时(→504)；可重试错误**指数退避+抖动**重试；信号量限制
在途上游并发；边缘中间件在途过载直接 **429** 快速失败。实测 MAX_INFLIGHT=2 下 8 并发 →
2×200 + 6×429。退避加抖动是为了避免"惊群"同步重试。

**④ 怎么知道 Agent 改坏没有？（评测的意义）**
把"期望行为"固化成用例自动判分。还讲一个真实细节：第一次评测暴露了我**自己的脆弱断言**
（"不支持5GHz"含子串"支持5GHz"误判）和 **LLM 裁判的局限**（不知真实时钟把工具返回的日期当
虚构）——说明评测不仅测被测对象，也逼着把测试本身写可靠。

## 4. 量化数据（讲的时候带上数字）

- 异步并发：5 并发 **3.8x** 加速（0.83s vs 3.16s）。
- 背压：MAX_INFLIGHT=2，8 并发 → 2 通过 / 6 个 429。
- 评测：7 条用例，工具选择 7/7，含 LLM 裁判 100% 通过。
- 离线测试：9 个用例，CI 每次 push 必过。
- RAG：知识库切 15 块，5GHz/保修等问题检索相关度 0.7–0.83。
- 可观测性：单轮 trace 可见 RAG 工具耗时 ~1.45s、整轮 ~3s、token ~1.3k。
- 嵌入式：BSP 组件 3 个、应用 4 个；双 I2S 端口 + DMA。

## 6. 高频追问与参考回答

- **"为什么不用 Arduino？"** Arduino 适合快速验证，我留了原型版讲演进；但面工程岗要体现
  寄存器/RTOS/构建系统，所以重写成 ESP-IDF。
- **"RAG 为什么不用 faiss/chroma？"** 知识库只有几十块，纯 Python 余弦足够且零依赖；
  数据量上万再换专用向量库——按规模选型，不过度工程。
- **"会话记忆怎么存？长了怎么办？"** 内存缓存 + SQLite 持久化，重启不丢；超过 N 轮做历史
  压缩（保留最近若干轮），防 token 超限和烧钱。
- **"并发上去了上游被限流怎么办？"** 见第 3 点四道防线。
- **"密钥怎么管？"** 只在服务端 `.env` / 运行时注入，绝不进设备/前端/镜像/git；`.gitignore`
  和 `.dockerignore` 都排除。
